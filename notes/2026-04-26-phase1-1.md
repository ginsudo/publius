### Retrieval scaffolding (commit `05ea3ed`)

- `data/eval/lib.ts` — env loader, Voyage HTTP client (`voyage-4-large`, batched 128/req), `node:sqlite` opener that auto-discovers `vec0.dylib`/`.so` from `data/eval/vendor/`, Federalist chunker producing 1,303 body-paragraph chunks + 71 footnote chunks (each header-prefixed with paper number / title / authors).
- `data/eval/build-index.ts` — drops/recreates tables, embeds in batches, idempotent.
- `data/eval/query.ts` — exports `queryIndex(question, k)` for reuse; CLI single-question wrapper.
- `data/eval/run.ts` — runs all active probes, writes `results.md`.
- `data/eval/README.md` — operational setup (Voyage key, sqlite-vec v0.1.6 binary fetch).
- Zero npm dependencies in the retrieval path. Node 22.5+ built-in `node:sqlite` + the loadable `sqlite-vec` extension binary, gitignored under `vendor/`.

### Bug fixes from first run (commit `d20abe1`)

- `node:sqlite` doesn't bind raw `ArrayBuffer` — wrap in `Uint8Array` (both `build-index.ts` and `query.ts`).
- `vec0` virtual table demands `SQLITE_INTEGER` for primary-key bindings; pass `BigInt(rowid)` to `insertVec`.

### Distance-metric switch (commit `6d83157`)

- `vec0` virtual table now declared with `distance_metric=cosine`. L2 produced identical rankings on normalized voyage-4-large vectors but uninterpretable scores; cosine reads directly as cosine similarity in `[-1, 1]`.

## Next logical slice — Phase 1.2 Q&A layer

The natural next step is to wrap `queryIndex()` with a Claude API call (Sonnet 4.6 per DECISIONS.md): pass top-K hits as structured context, ask the model the user's question, and have it cite back to the chunks it used. Two things to design before any code:

1. **The system prompt.** Per CLAUDE.md, the highest-stakes artifact in the project. Owner sets it; Claude Code drafts and tests against 10–15 questions spanning easy and hard cases, including ones where genuine disagreement should surface and a naive model would flatten. Phase 1.1 only exercises retrieval — surfacing argument vs. argument disagreement (Hamilton vs. Madison) is a Phase 1.3 hardening concern, but the system prompt has to be drafted to know what to harden.
2. **The HTTP boundary.** The first natural moment to introduce Next.js — an API route at `/app/api/ask` that takes a question, calls `queryIndex()`, calls Claude, returns the answered text + citations. Still no UI. The Phase 1.4 work (Vercel deploy + observability) follows from this.
